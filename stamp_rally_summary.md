# 春日市 デジタルスタンプラリー プロジェクト検討まとめ

作成日：2026年5月8日

---

## 1. プロジェクト概要

春日市が公開する6コースのウォーキングマップを活用し、地域イベント実行委員会が主体となってデジタルスタンプラリーを実施するプロジェクト。将来的には自治体との連携協定を締結し、他自治体への横展開も見据えたプラットフォームとして発展させることを目指す。

| 項目 | 内容 |
|---|---|
| 運営主体 | 地域イベント実行委員会 |
| 対象エリア | 福岡県春日市 |
| 参加想定規模 | 中規模（100〜1,000人） |
| 利用形態 | アプリ不要・Webブラウザのみで完結 |
| 開催時期 | 数ヶ月以内に試験運用開始 |
| 試験運用方針 | まず一部コースで試験的に実施 |

## 2. 春日市ウォーキングマップ 既存資産（6コース）

春日市公式サイト（令和7年3月更新）で公開中のPDFマップをコース基盤として活用。

| # | コース名 | 距離 | 所要時間 | 特徴 |
|---|---|---|---|---|
| 1 | JR南福岡駅〜春日駅 | 約13km | 180分 | 長距離・複合ルート（試験運用では除外候補） |
| 2 | 春日公園〜九大筑紫キャンパス | 約5km | 60分 | 公園・大学 |
| 3 | 春日市役所〜JR南福岡駅 | 約4km | 50分 | 市街地・交通拠点 |
| 4 | 日拝塚古墳〜白水大池公園 | 約4km | 50分 | 史跡・自然 |
| 5 | 奴国の丘歴史資料館〜遺跡めぐり | 約5km | 70分 | 歴史・遺跡 |
| 6 | 岡本遺跡〜都市高速 | 約6km | 80分 | 遺跡・インフラ景観 |

**活用ポイント**
- 各コース上の立ち寄りスポットをそのままスタンプポイントに転用可能
- 既存PDFマップをデジタルマップへの変換素材として利用
- コース単位で「1コース達成」「全6コース制覇」など段階的な達成感を設計できる
- 試験運用はコース2〜6から1〜2コースを選定予定

## 4. 独自開発 要件定義

### 4-1. ユーザー種別

| 種別 | 説明 |
|---|---|
| 参加者 | スタンプを集める一般ユーザー（匿名セッション可） |
| スポット管理者 | 各チェックポイントの店舗・施設担当者 |
| 運営管理者 | 実行委員会スタッフ（管理画面アクセス） |

### 4-2. 参加者向け機能

**Phase1（MVP・必須）**
- コース一覧・詳細表示
- スポットマップ表示（Google Maps埋め込みまたはLeaflet）
- QRコード読み取りによるスタンプ取得
- スタンプ帳画面（取得済み・未取得の一覧）
- コース完走判定・達成バッジ表示
- 景品応募フォーム（スタンプ数条件クリア後に解放）

**Phase2（拡張機能）**
- GPS自動チェックイン（スポット半径内で自動取得）
- コース上のナビ機能
- SNSシェア機能（達成画面）
- 多言語対応（日英）

### 4-3. スポット管理者向け機能
- QRコード発行・印刷
- 自スポットのチェックイン数確認

### 4-4. 運営管理者向け機能
- コース・スポットのCRUD管理
- 参加者数・スタンプ取得数のダッシュボード
- 景品応募者一覧・当選者管理
- QRコード一括生成・出力
- イベント期間設定（開始・終了日時）
- データエクスポート（CSV・PDF）

### 4-5. 認証・セキュリティ

| 対象 | 方式 |
|---|---|
| 参加者 | 匿名セッション（メアドorニックネームのみ、ログイン不要） |
| スポット管理者 | ID/PW ログイン |
| 運営管理者 | ID/PW ログイン ＋ 管理画面アクセス制限 |

不正防止：同一セッションからの同スポット二重取得はブロック。

### 4-6. 主要データ設計

```
コース (courses)
  └ スポット (spots)  ← QRコード紐づき

参加者セッション (participants)
  └ スタンプ取得履歴 (stamp_logs)
      └ 取得日時・スポットID・位置情報(任意)

景品応募 (prize_applications)
イベント設定 (event_settings)
```

## 5. マルチテナント・拡張設計（3点セット）

### 5-1. マルチテナント設計

**採用方式：スキーマ分離（方式B）**
同一DBでテナントごとにスキーマ（名前空間）を分ける。コスト・管理性・セキュリティのバランスが良く中規模向き。

```
schema: global
  tenants        -- 自治体・テナント管理（slug, logo, color, domain, contract_type）
  tenant_admins  -- 運営管理者アカウント

schema: tenant_kasuga   -- 春日市
  courses / spots / events / participants / stamp_logs / prize_applications

schema: tenant_dazaifu  -- 太宰府市（将来追加）
  （同構造）
```

**重要：DB設計だけは最初からマルチテナント対応にしておくこと。後からのスキーマ移行は工数大。**

### 5-2. データエクスポート

管理画面から自治体担当者がいつでもDL可能。自治体システムとのAPI連携口も当初から設計。

- 参加者統計サマリー（PDF/CSV）
- スポット別訪問数（CSV）
- コース別完走率（CSV）
- 時間帯別アクセス分析（CSV）
- 景品応募者一覧（CSV）

**外部連携API（自治体システム向け）**
```
GET /api/v1/stats/summary        -- 参加者数・スタンプ総数・完走者数
GET /api/v1/stats/spots          -- スポット別訪問数ランキング
GET /api/v1/stats/courses        -- コース別参加者数・完走率
GET /api/v1/participants/{id}/stamps  -- 個人のスタンプ取得履歴（健康ポイント連携用）
```

APIキーはテナントごとに発行。

### 5-3. ホワイトラベル対応

テナントごとにカスタマイズできる要素：
- ロゴ・カラー（CSS変数で動的切り替え）
- サブドメイン：`kasuga.stampapp.jp` / `dazaifu.stampapp.jp`
- 自治体独自ドメインのCNAME対応（`stamp.city.kasuga.fukuoka.jp`）
- フッター・利用規約・プライバシーポリシーもテナント別設定

## 6. 技術スタック

| レイヤー | 採用技術 | 理由 |
|---|---|---|
| フロントエンド | Next.js + Tailwind CSS | SSR・パフォーマンス・開発効率 |
| バックエンド | Next.js API Routes / FastAPI | 統合開発・軽量運用 |
| DB | PostgreSQL（Supabase） | スキーマ分離・低コスト・無料枠あり |
| 地図 | Google Maps API / Leaflet | 柔軟性・OSSコスト削減 |
| QR生成 | qrcode.js | 軽量・ライセンスフリー |
| ホスティング | Vercel + Supabase | 小規模なら無料枠で運用可 |
| メール送信 | Resend / SendGrid | 景品応募通知等 |

## 8. 自治体連携・企業連携協定 戦略

### 8-1. 連携の段階イメージ

| フェーズ | 時期 | 状態 | 主なアクション |
|---|---|---|---|
| フェーズ1 | 〜3ヶ月 | 試験運用 | 地域実行委員会で独自開発・データ収集 |
| フェーズ2 | 3〜6ヶ月 | 効果測定 | 参加者数・完走率等のエビデンス整理 |
| フェーズ3 | 6ヶ月〜1年 | 連携協定交渉 | 自治体へ非公式提案→協定締結 |
| フェーズ4 | 1年〜 | 公認化・展開 | 予算づけ・他コース・他自治体へ横展開 |

### 8-2. 自治体側のメリット（提案資料の核）

- 健康増進施策のDX化：ウォーキングデータ・参加者統計の提供
- 既存ウォーキングマップの活性化・デジタル化
- 観光・回遊促進のエビデンス（コース別・スポット別行動データ）
- 低コストで継続運用できるプラットフォームとして市民サービス充実
- SNSシェア・口コミ拡散による広報強化

### 8-3. 連携協定締結に向けた準備事項

- **実績づくり**：試験運用のデータ（参加者数・完走率・スポット訪問数）を蓄積
- **法人格の整理**：一般社団法人・NPO法人の設立、または既存法人との連携
- **個人情報保護**：取得情報を最小限にし、将来のセキュリティ要件に備える
- **データ提供API**：当初から自治体システム連携の口を設計に組み込む
